The Problem

Cloud bills are designed
to be complex

Cloud providers charge for hundreds of individual services across dozens of regions. A typical company runs thousands of these resources — and many of them are quietly billing every month for work they no longer do.

A virtual machine provisioned for a test last year? Still running. A database snapshot from before the last migration? Still stored. A reserved IP address no longer in use? Still billing. These add up fast.

The industry average is that 1 in every 3 dollars spent on cloud infrastructure goes to unused resources. AI Cloud Cost Detective identifies every one of them across AWS, Azure, and GCP in a single scan.

The Process

Four steps from login
to cost savings

Architecture

How the system is built

Everything runs inside your own environment — your browser talks to a backend that only your team can reach. Your cloud credentials never leave your infrastructure.

⚛️
React 18 Frontend
TypeScript + Vite + Tailwind CSS. Source maps disabled. Security headers enforced by nginx. No external CDN dependencies.
⚡
FastAPI Backend
Python 3.11. Async throughout. JWT auth with JTI-based token revocation. Rate limiting on every public endpoint. Runs as non-root user.
☁️
Cloud Scanners
Three dedicated modules — boto3 (AWS, 87 services), azure-mgmt (Azure, 20 services), google-api-python-client (GCP, 18 services). Only requested services are scanned.
🧠
AI Analyser
14 providers supported. Auto-detects the first available API key. Falls back to built-in rule engine if no key is configured — zero cost, zero config.
💾
PostgreSQL 16
Stores analyses, users, and revoked tokens. Analyses auto-purged after 2 days. Users can delete individual entries. In-memory fallback when no DB is configured.
🐳
Docker Compose
Four containers — frontend, backend, database, and a Cloudflare tunnel for instant public access. One command starts everything. Backend and database ports are not exposed publicly.
⚙️
GitHub Actions CI
Automated pipeline on every push and pull request: backend pytest against real PostgreSQL 16, TypeScript type-check, frontend vitest, and Docker build validation.

Security Architecture

Built with security
as a requirement

Your cloud credentials are used to make API calls and are never stored in the database. The application runs entirely within your own infrastructure.

🔐
JWT + Token Revocation
Stateless sessions with instant invalidation

JWTs are signed with HS256 using a secret of at least 32 characters. Tokens expire after 8 hours. Every request passes through _verify_token before any logic runs.

In this app: Each token carries a unique JTI (JWT ID). Logout stores the JTI in the revoked_tokens table — the token is dead instantly, not just after expiry. Expired JTIs are purged automatically every 12 hours.
🛡️
Input Validation
Validate all data at the boundary

Unvalidated input enables injection vulnerabilities and data errors. Validating shape, format, and range at the API boundary means only well-formed data reaches the database or cloud SDKs.

In this app: All request bodies parsed through Pydantic models. Emails validated with regex. Passwords 8–128 chars. AWS account IDs must be exactly 12 digits. Azure Subscription IDs must be UUIDs. GCP Project IDs must match GCP naming rules. Invalid input returns 422 before any logic runs.
🚦
Rate Limiting
Cap request frequency per IP and email

Without rate limits, automated tools can attempt millions of password combinations or register thousands of accounts in minutes.

In this app: Sliding-window in-memory buckets per (IP, action). Login: 20 req/min per IP, 10/min per email. Signup: 10 per 5 min per IP. Password change: 5 per 5 min per user. Concurrent scans: max 5 platform-wide, max 3 per user. Exceeding any limit returns 429.
📋
Security Headers
Browser-enforced protection policies

HTTP response headers instruct browsers to enforce security policies — preventing clickjacking, MIME sniffing, and injection of external scripts.

In this app: nginx adds on every response:
• X-Frame-Options: DENY — blocks clickjacking
• X-Content-Type-Options: nosniff
• X-XSS-Protection: 1; mode=block
• Referrer-Policy: strict-origin-when-cross-origin
• Permissions-Policy — disables geolocation, mic, camera
• CSP — scripts and styles from self only
✅
Pre-scan Validation
Verify credentials before touching anything

Starting a full scan with incorrect credentials wastes minutes. Pre-scan validation makes a single lightweight API call to confirm credentials are valid before any real work starts.

In this app: POST /api/validate runs a provider-specific preflight:
• AWS: STS GetCallerIdentity
• Azure: Subscription object fetch
• GCP: Cloud Resource Manager project check
All raw SDK exception text is stripped — only a safe message is returned.
🐳
Container Isolation
Least-privilege process isolation

Docker containers run as root by default. Running as a non-privileged user limits the blast radius of any security incident and is a container best practice.

In this app: The backend Dockerfile creates a dedicated appuser:appgroup account, transfers file ownership, then switches to it with USER appuser before the server starts. Backend and database ports are bound to 127.0.0.1 only — not exposed to the internet. Source maps disabled; .map and .ts file routes blocked by nginx.
